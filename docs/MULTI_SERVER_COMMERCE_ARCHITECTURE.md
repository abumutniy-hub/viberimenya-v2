# Архитектура сайта, Telegram и MAX на двух серверах

## Цель

Российский сервер является единственным торговым ядром. На нём работают сайт, CRM, MAX, API, PostgreSQL, платёжный модуль, остатки, промокоды, бонусы и журнал событий. Текущий сервер обслуживает Telegram и обращается к торговому ядру только через защищённый API.

Нельзя создавать две независимые базы заказов или разрешать Telegram и MAX напрямую изменять одни и те же таблицы: это приводит к дублям, расхождению остатков и повторной обработке платежей.

## Целевая схема

```text
Клиент сайта ─┐
MAX ──────────┼─> Российский сервер: Commerce Core API ─> PostgreSQL
CRM ──────────┤                 │                    ├─> ЮKassa
Telegram ─────┘                 ├─> Outbox           └─> хранилище файлов
 (текущий сервер, адаптер) <────┘
```

## Границы ответственности

- Commerce Core единолично создаёт и изменяет заказ, рассчитывает сумму, резервирует остатки, применяет промокод и бонусы, создаёт платёж и принимает webhook.
- Telegram и MAX являются каналами общения. Они передают команды в Commerce Core и показывают полученный результат.
- PostgreSQL имеет одного владельца записи и не реплицируется в режиме multi-master.
- Все исходящие уведомления сначала фиксируются в outbox. Повторная доставка не создаёт новое бизнес-событие.
- Фотографии хранятся в общем объектном хранилище или на российском сервере и выдаются по HTTPS; локальный диск Telegram-сервера не является главным хранилищем.

## Защищённое соединение

- Между серверами используется HTTPS, желательно поверх WireGuard.
- Для каждого канала создаётся отдельный длинный сервисный секрет: Telegram и MAX не делят один ключ.
- Секреты передаются только в заголовках, хранятся в `.env` с правами `600` и регулярно меняются.
- API проверяет канал, секрет, допустимую операцию и ограничивает частоту запросов.
- Платёжный webhook доступен только на Commerce Core. Перед применением события Commerce Core повторно получает платёж из API ЮKassa и сверяет магазин, заказ, сумму и валюту.

## Подготовленная совместимость

Telegram использует `COMMERCE_CORE_URL` и `COMMERCE_TELEGRAM_SERVICE_TOKEN`. Пока процессы находятся на одном сервере, сохраняется прежний локальный режим. Удалённый доступ становится возможен только после явного задания отдельного сервисного секрета на обоих серверах.

## Порядок переноса без простоя

1. Подготовить российский сервер, PostgreSQL, резервное копирование, мониторинг и закрытую сеть между серверами.
2. Развернуть тот же release Commerce Core, не переключая публичный домен и не включая платежи.
3. Перенести копию базы и файлов, проверить контрольные суммы, миграции и восстановление backup.
4. На короткое окно обслуживания остановить запись заказов, выполнить финальную синхронизацию и назначить российскую базу единственным источником данных.
5. Переключить сайт/CRM и webhook ЮKassa на российский сервер.
6. Настроить текущий Telegram-сервер на новый `COMMERCE_CORE_URL` и отдельный сервисный секрет.
7. Проверить один заказ от каждого канала, повтор webhook, истечение оплаты и отсутствие дублей.
8. После стабильного наблюдения удалить прямой доступ Telegram-процесса к торговым таблицам. На переходном этапе его нельзя расширять новыми прямыми SQL-операциями.

## Следующее усиление

- Подпись каждого межсерверного запроса HMAC с timestamp и nonce защищает от повторной отправки перехваченного запроса.
- Отдельный `channel_inbox` с уникальным внешним ID делает повторные команды Telegram/MAX идемпотентными.
- Correlation ID связывает заказ, платёж, webhook и уведомления в одном журнале.
- Автоматический circuit breaker временно прекращает команды записи при недоступности Commerce Core, не создавая локальные «временные» заказы.
- Регулярная проверка восстановления из backup обязательна: наличие файла backup без восстановления не считается защитой.
