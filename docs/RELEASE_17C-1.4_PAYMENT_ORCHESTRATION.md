# 17C-1.4 PAYMENT ORCHESTRATION + MULTICHANNEL PAYMENT LINK

## Цель

После подтверждения заказа менеджером автоматически создать или переиспользовать платёж ЮKassa, показать клиенту кнопку оплаты на странице заказа и отправить ссылку в подключённые каналы.

## Исправленная причина

Ручной способ `transfer_after_confirm` отображался первым и становился значением по умолчанию даже при включённой ЮKassa. Автоматическое создание платежа корректно работало только для `online_card` и `sbp`, поэтому тестовый заказ оставался в ручном сценарии.

## Изменения

- `online_card` и `sbp` идут первыми в web checkout, когда онлайн-оплата доступна;
- подписи явно различают автоматическую ЮKassa и ручной перевод по реквизитам;
- Telegram checkout показывает онлайн-оплату первой;
- подтверждение уже подтверждённого онлайн-заказа безопасно повторяет идемпотентную подготовку платежа;
- CRM получила действие `Создать оплату ЮKassa`;
- старый подтверждённый заказ с `transfer_after_confirm` можно безопасно перевести на `online_card` и создать платёж;
- ручное поле ссылки осталось только у ручного перевода;
- страница заказа проверяет появление ссылки/статуса каждые 5 секунд, пока ожидается онлайн-оплата;
- Telegram получает кнопку `Оплатить заказ` через существующий outbox;
- MAX customer notifications получили transport через официальный `POST https://platform-api2.max.ru/messages` и link-кнопки;
- MAX transport активируется только при наличии `MAX_BOT_TOKEN` и включённых feature flags `maxEnabled` + `maxNotificationsEnabled`;
- webhook ЮKassa остаётся источником истины для статуса `paid`, бонусов и уведомления об оплате.

## Без изменений

- миграция БД не требуется;
- схема платежей, webhook, возвраты и бонусная логика не меняются;
- MAX authentication из 17C-1.3 не изменяется;
- MAX notifications не включаются автоматически и не требуют добавления секрета в release payload.
